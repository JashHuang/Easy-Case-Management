# AI Agent Workflow

Version: 1.1  
Last Updated: 2026-03-06

## 1. Purpose

This document defines the target AI workflow for converting uploaded evidence files into structured timeline events in Easy Case Management.

Note: AI workflow is currently disabled while the architecture is being redesigned.

This workflow is aligned with the current system baseline:
- Frontend: React + Vite
- Backend: Supabase (Auth, Postgres, Storage)
- Existing tables: `cases`, `events`, `attachments`, `profiles`

AI output is never written directly to production tables without system validation and human review.

## 2. Scope

In scope (AI MVP):
- Parse uploaded files (PDF/image/doc/text)
- Extract candidate events
- Build normalized timeline draft
- Human review and approval
- Insert approved events into `events`

Out of scope (post-MVP):
- knowledge graph
- RAG chat
- fully autonomous write-back without review

## 3. End-to-End Flow

```text
Upload Attachment
  -> Store in Cloudflare R2 + attachments record
  -> Trigger AI job (manual or async)
  -> Parse document
  -> Extract candidate events (LLM)
  -> Normalize + validate
  -> Save draft to ai_results
  -> Human review UI
  -> Approve
  -> System writes to events + updates attachments status
```

## 4. Agent/Service Responsibilities

The implementation can be one service with modular steps or multiple workers. Responsibilities remain the same.

### 4.1 Ingestion
- Trigger condition: new/updated attachment requiring AI analysis
- Required input: `attachment_id`, `case_id`, `file_url|storage_path`, `file_type`
- Output: queued processing job

### 4.2 Document Parser
- Converts source file to plain text with page/segment metadata
- Supports: PDF, image (OCR), DOC/DOCX, TXT
- Output:

```json
{
  "text": "...",
  "segments": [{ "page": 1, "text": "..." }]
}
```

### 4.3 Information Extraction
- Uses LLM to extract candidate timeline events
- Must return strict JSON schema
- Output (example):

```json
{
  "summary": "...",
  "events": [
    {
      "event_date": "2025-01-10",
      "title": "Meeting held",
      "description": "...",
      "source_ref": "file.pdf#p2",
      "confidence": 0.83
    }
  ]
}
```

### 4.4 Timeline Normalizer
- Normalizes date formats
- Sorts events by `event_date`
- Marks duplicate/similar events for review
- Flags uncertain dates as `needs_review`

### 4.5 Validation
- Schema validation
- Content checks (required fields, length, date validity)
- Confidence threshold policy (for example `< 0.5` => always manual review)

### 4.6 Persistence Mapper + Writer
- Converts validated draft into approved DB write payload
- Writes only through Supabase client/API
- Never executes raw SQL generated by LLM

## 5. Data Contracts

## 5.1 `attachments` status lifecycle (new fields)

Recommended statuses:
- `uploaded`
- `pending_ai`
- `processing`
- `review_ready`
- `approved`
- `rejected`
- `failed`

State transitions must be explicit and auditable.

## 5.2 `ai_results` table (new)

Purpose: store AI draft and review state before writing to `events`.

Suggested columns:
- `id` (uuid, pk)
- `attachment_id` (uuid, fk -> attachments.id)
- `case_id` (uuid, fk -> cases.id)
- `raw_output` (jsonb)
- `normalized_output` (jsonb)
- `review_status` (`pending` | `approved` | `rejected`)
- `model_name` (text)
- `avg_confidence` (numeric)
- `error_message` (text, nullable)
- `created_at`, `updated_at`

## 5.3 `events` write rules

Approved AI events map to existing `events` columns:
- `case_id`
- `title`
- `description`
- `event_date`

If source tracking is required, add nullable columns in migration (for example `source_ref`, `ai_confidence`).

## 6. Human Review Requirements

Review UI must support:
- edit candidate event
- delete candidate event
- resolve duplicate suggestions
- approve/reject all or selected events

Human approval is mandatory for AI MVP.

## 7. Error Handling and Retry

Error categories:
- parsing failure
- model timeout/failure
- invalid JSON output
- validation failure

Policy:
- bounded retry (for example max 3)
- permanent failure sets `attachments.status = failed`
- failure reason recorded in `ai_results.error_message`

## 8. Security and Compliance

Mandatory rules:
- AI output is treated as untrusted input
- All writes pass validation and authorization checks
- RLS remains enforced for all read/write paths
- Secrets (LLM keys) stay server-side only

## 9. Observability

Track at minimum:
- job latency (parse/extract/total)
- retry count
- failure rate by file type
- manual correction ratio

These metrics drive prompt and pipeline quality improvements.

## 10. AI MVP Acceptance Criteria

AI MVP is acceptable when:
- user can trigger AI analysis from attachment context
- parsed/extracted draft is visible in review UI
- approved events are written correctly to `events`
- failed jobs are visible and retryable
- no direct LLM output bypasses validation/review
